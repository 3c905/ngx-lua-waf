# ngx-lua-waf 与主流 WAF 深度对比分析

## 一、本项目架构总览

```
Nginx Access Phase        Nginx Body Filter Phase
       │                           │
       ▼                           ▼
   waf.lua ────────────►    response.lua
       │
       ├──► whiteip()      ── IP 白名单
       ├──► blockip()      ── IP 黑名单（精确匹配）
       ├──► methodcheck()  ── TRACE/TRACK 拦截
       ├──► traversal()    ── 路径穿越/空字节
       ├──► headers()      ── Header 攻击检测
       ├──► denycc()       ── CC 防护（计数器式）
       ├──► 扫描器特征头   ── Acunetix/X-Scan
       ├──► referer()      ── 恶意 Referer
       ├──► whiteurl()     ── URL 白名单
       ├──► ua()           ── User-Agent 黑名单
       ├──► dangerous()    ── 敏感路径/文件
       ├──► url()          ── URL 黑名单
       ├──► args()         ── GET 参数检测
       ├──► cookie()       ── Cookie 检测
       └──► POST 检测      ── multipart/表单
```

**规则引擎**: 纯 PCRE 正则线性匹配，每次请求从磁盘读取规则文件。
**CC 引擎**: 基于 `ngx.shared.limit` 的简单计数器，`key = IP + URI`。
**日志**: 同步文件追加，无轮转/限流。

## 三、优缺点详析

### ✅ 优点

1. **极致轻量**
   - 核心代码仅 ~500 行 Lua，无外部依赖
   - 内存占用低，适合资源受限环境

2. **OpenResty 原生集成**
   - 直接运行在 Nginx worker 进程内，无网络跳转开销
   - 延迟增加通常 < 1ms（不含规则 IO 开销）

3. **规则热更新**
   - 修改 `wafconf/` 规则文件即时生效，无需 reload
   - 适合快速响应新攻击特征

4. **技术栈分组规则**
   - `dangerous` 规则按 `[core]`/`[java]`/`[nodejs]`/`[php]`/`[aggressive]` 分组
   - 场景化开关，降低误伤

5. **双向检测**
   - 请求阶段（access）+ 响应阶段（body_filter）联动
   - 响应泄露检测是多数开源 WAF 缺失的能力

6. **结构化日志**
   - 命中标记包含类型、技术栈、状态码、规则原文
   - 便于日志分析和告警

### ❌ 缺点

#### 1. 性能瓶颈：每次请求读取规则文件
```lua
-- init.lua 第53行
function read_rule(var)
    file = io.open(rulepath..'/'..var,"r")  -- 每次请求都打开文件！
    -- ...
end

-- waf.lua 加载时调用
urlrules=read_rule('url')  -- 每次 worker 加载时读一次？不，看 init.lua 的加载位置
```

**实际行为**: `init.lua` 中的 `urlrules=read_rule('url')` 等是在 `require 'init'` 时执行，即 worker 启动时加载一次。但每次请求都会 `require 'init'`，在 Lua 中 `require` 有缓存，所以实际上每个 worker 只加载一次。

**但是**: `init.lua` 没有使用 `local` 修饰全局变量（`urlrules`、`argsrules` 等），在多 worker 环境下，虽然 Lua 是单线程的，但全局变量的访问性能差于局部变量。更重要的是，**规则无法热更新**——因为 `require` 缓存，`read_rule` 只在第一次 `require` 时执行。

> README 声称"热更新无需 reload"，但实际上 `require` 有缓存，修改规则文件后不会自动生效，除非清理 package.loaded。这是一个文档与实现不符的问题。

#### 2. CC 防御缺陷（核心问题）

| 缺陷 | 影响 |
|------|------|
| **算法简陋** | 简单计数器，非 Token Bucket/滑动窗口，突发流量无法平滑处理 |
| **Key 设计缺陷** | `IP + URI` 意味着每个 URI 独立计数。攻击者用随机参数 `?r=xxx` 即可绕过 |
| **无静态文件区分** | JS/CSS/图片请求与 API 请求共用同一策略，静态文件无法有效防护 |
| **无渐进惩罚** | 只返回 503，不封禁 IP，攻击者持续重试无成本 |
| **无 Bot 识别** | 无法区分搜索引擎爬虫、正常用户、攻击脚本 |
| **无分布式** | 单机 `ngx.shared.dict`，多节点部署时 CC 计数不共享 |
| **检测顺序问题** | `denycc()` 在 `whiteurl()` 之前，白名单 URL 仍会被 CC 拦截 |

#### 3. 规则引擎能力不足

| 问题 | 说明 |
|------|------|
| 无规则 ID | 无法精确定位和管理单条规则 |
| 无规则优先级 | 所有规则平等匹配，无法设置例外 |
| 无规则动作 | 只能 block/log，无法 allow/skip/chain |
| 无变量提取 | 无法将匹配结果用于后续规则 |
| 无解码链 | 不支持 `urlDecode → base64Decode → lowercase` 这样的处理链 |
| 无请求体解析 | multipart 解析手写，不支持 JSON/XML 请求体检测 |

#### 4. 其他安全缺陷

| 问题 | 风险 |
|------|------|
| 无请求体大小限制 | 超大 POST 可导致内存/CPU 耗尽 |
| 日志无保护 | 攻击者可构造大量命中日志填满磁盘 |
| IP 黑白名单无 CIDR | 无法配置 `10.0.0.0/8` 这样的网段 |
| 无 X-Forwarded-For 解析 | 在 CDN/负载均衡后无法获取真实 IP |
| 无地理限制 | 无法按国家/地区封禁 |
| 安装脚本过时 | Nginx 1.2.4 (2012)、LuaJIT 2.0.0、lua-nginx-module 0.7.4 均存在已知漏洞 |

## 五、CC 防御专项增强设计

### 5.1 用户需求的准确理解

> "CC 的防御希望是对静态文件的过滤后进行抵御"

**解读**: 需要建立"请求分类 → 差异化策略"的 CC 防御体系：

1. **先识别请求类型**（静态文件 / 动态请求 / API / 上传）
2. **对不同类型应用不同 CC 策略**
3. **对静态文件实施专门防护**（因为静态文件 CC 攻击有独特特征）

### 5.2 主流 WAF 的 CC 防御策略对比

| WAF | 静态文件 CC 策略 | 动态请求 CC 策略 | Bot 识别 |
|-----|------------------|------------------|----------|
| **Cloudflare** | 边缘缓存优先，回源限制；无 Referer/Cookie 的请求严格限速 | 基于 Bot Score 的渐进式挑战 | JS Challenge + ML |
| **AWS WAF** | Rate-based rule 可排除静态路径；CloudFront 缓存层吸收 | Token Bucket，支持 IP + Header 组合 | CAPTCHA (第三方) |
| **阿里云** | 资源类型自动识别，静态资源使用宽松阈值 | 行为分析模型，API 级细粒度控制 | 设备指纹 + 深度学习 |
| **ModSecurity** | 依赖规则配置 `SecRule REQUEST_FILENAME` 排除 | `SecAction` + `initcol` 集合计数 | 无内置 |

### 5.3 建议的防御策略（本项目补强方向）

    │

### 5.4 静态文件 CC 的特殊考虑

静态文件请求的特征和攻击模式：

| 正常特征 | 攻击特征 |
|----------|----------|
| 有 Referer（来自同站页面） | 无 Referer 或伪造 |
| 有合理 UA（浏览器特征） | 无 UA 或明显工具特征 |
| Accept 头包含 MIME 类型 | Accept: */* |
| 请求顺序符合页面加载逻辑 | 随机顺序、并发极高 |
| Cookie 通常携带（同域） | 无 Cookie |

**防御建议**:
1. **缓存层优先**: 静态文件应由 CDN/Nginx 缓存吸收，WAF 只处理回源请求
2. **Referer 验证**: 对无 Referer 的静态文件请求降低阈值
3. **并发连接限制**: 单 IP 并发连接数限制（Nginx `limit_conn` 层）
4. **带宽型 CC**: 对大文件（视频/图片）设置更低阈值
5. **指纹验证**: 对可疑请求返回 Set-Cookie，后续请求验证
